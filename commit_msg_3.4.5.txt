feat(web-dashboard): implement payout approval interface (Task 3.4.5 ✅)

COMPREHENSIVE DUAL APPROVAL WORKFLOW IMPLEMENTATION

This commit implements a complete payout approval interface for the Digital Stokvel Web Dashboard, enabling Treasurers to review and approve/reject payouts initiated by Chairpersons. The implementation follows the dual approval workflow specified in design.md Section 2.2.3 and Payout API design in Section 5.4.

═══════════════════════════════════════════════════════════════════════

🎯 KEY FEATURES

1. **Dual Approval Workflow**
   - Chairperson initiates payout → pending_treasurer_approval
   - Treasurer reviews with full payout details
   - Treasurer approves with PIN + optional comment → approved → processing
   - Treasurer can reject with PIN + required reason → rejected
   - 7-state status flow: pending_chairperson_approval, pending_treasurer_approval, approved, processing, completed, failed, rejected

2. **Payout Type System**
   - Rotating payouts: Sequential member rotation
   - Year-end payouts: End-of-year pot distribution
   - Emergency payouts: Urgent disbursements
   - Type-specific visual indicators with emoji badges (🔄 rotating, 🎉 year_end, 🚨 emergency)
   - Color-coded type badges: rotating=blue, year_end=purple, emergency=red

3. **Security & Authorization**
   - PIN confirmation required for all approve/reject actions
   - 4-digit PIN input with password masking and monospace font
   - Role-based access control: canApprovePayout() checks user role vs payout status
   - Audit trail: Comments for approvals, reasons for rejections
   - JWT authentication integration with protected routes

4. **Comprehensive Status Filtering**
   - Pending My Approval: Shows payouts awaiting user's action
   - Approved / Processing / Completed: Track payout lifecycle
   - Rejected: Review declined payouts
   - All: Full payout history
   - Real-time status badge updates with 7 color variants

5. **Dual UI Layout**
   - Card-based grid for pending approvals (action-oriented, better UX for approval workflow)
   - Table view for all payouts (data-dense, better for historical review)
   - Pending cards show: type badge, status badge, amount, initiator info, date, note
   - Table columns: Type, Amount, Status, Initiated By, Initiated At, Approved At

6. **Approval/Rejection Forms**
   - Approval form: PIN input + optional comment textarea
   - Rejection form: PIN input + required reason textarea
   - Inline form expansion on card click (Approve/Reject buttons)
   - Form validation: PIN required (4 digits), rejection reason required
   - Confirm/Cancel buttons for form submission
   - Success/error messaging with automatic dismissal

3. **Audit Trail**
   - Comments captured for approvals (optional, for notes)
   - Reasons captured for rejections (required, for accountability)
   - Approval timestamps tracked for compliance

4. **JWT Authentication**
   - All API calls include JWT token in Authorization header
   - Protected routes enforce authentication before access
   - Automatic redirect to login on unauthorized access

📊 API INTEGRATION

**Payout Service Endpoints:**

1. **GET /v1/payouts**
   - List payouts with filters: group_id, status, from_date, to_date
   - Returns: Paginated list with page, page_size, total_count
   - Used for: All payouts table view

2. **GET /v1/payouts/:id**
   - Get single payout details by ID
   - Returns: Complete payout with disbursements, initiator, approver
   - Used for: Detailed payout review before approval

3. **POST /v1/payouts**
   - Initiate new payout (Chairperson only)
   - Body: { group_id, type, amount, note, recipients[] }
   - Returns: Created payout with pending_treasurer_approval status

4. **POST /v1/payouts/:id/approve**
   - Approve payout (Treasurer only)
   - Body: { pin, comment? }
   - Returns: Updated payout with approved status

5. **POST /v1/payouts/:id/reject**
   - Reject payout (Treasurer only)
   - Body: { pin, reason }
   - Returns: Updated payout with rejected status

**Request/Response Format:**

Approve Request:
```json
{
  "pin": "encrypted_pin",
  "comment": "Approved - all contributions received"
}
```

Reject Request:
```json
{
  "pin": "encrypted_pin",
  "reason": "Insufficient funds for proposed emergency payout"
}
```

🎨 UI/UX DESIGN DECISIONS

1. **Card-Based Layout for Pending Approvals**
   - Rationale: Better UX for action-oriented workflow
   - Each card is self-contained with approve/reject actions
   - Visual hierarchy: Type badge → Amount → Status → Initiator → Actions
   - Hover effects for interactivity feedback

2. **Table Layout for Historical Payouts**
   - Rationale: Better data density for review
   - Sortable columns for flexible data exploration
   - Responsive with horizontal scroll on narrow viewports
   - Sticky header for context during scroll

3. **Purple Gradient for Payout Button**
   - Rationale: Visual differentiation from other buttons
   - Green used for members, blue for contributions
   - Purple conveys importance and authority (treasurer role)
   - Gradient: #9C27B0 → #E91E63 (90deg linear)

4. **Inline Form Expansion**
   - Rationale: Reduce context switching
   - Forms appear in-place on card when Approve/Reject clicked
   - User maintains context of payout being acted upon
   - Cancel returns to card view without page navigation

5. **Type Badges with Emoji Indicators**
   - 🔄 Rotating: Sequential member payouts (blue)
   - 🎉 Year-end: Annual pot distribution (purple)
   - 🚨 Emergency: Urgent disbursements (red)
   - Rationale: Quick visual recognition of payout purpose

6. **7-State Status Tracking**
   - pending_chairperson_approval: Orange (awaiting chairperson)
   - pending_treasurer_approval: Blue (awaiting treasurer)
   - approved: Green (approved, awaiting processing)
   - processing: Blue (disbursement in progress)
   - completed: Green (funds transferred)
   - failed: Red (processing error)
   - rejected: Red (approval declined)
   - Rationale: Comprehensive lifecycle visibility
