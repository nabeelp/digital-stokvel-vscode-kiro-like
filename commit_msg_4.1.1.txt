feat(infrastructure): implement CBS REST API client with resilience patterns (Task 4.1.1 ✅)

Implements synchronous REST API client for Core Banking System (CBS) integration
with production-ready resilience patterns using Polly library. This provides the
foundation for all CBS operations including account creation, transactions, and
balance inquiries.

**Implementation:**

1. **CBS Client (CbsClient.cs - ~450 lines):**
   - Implements ICbsClient interface with 5 API operations
   - Uses HttpClient with typed client pattern via dependency injection
   - Configuration via IOptions<CbsClientOptions> (BaseUrl, ApiKey, Timeout)
   - Comprehensive structured logging with ILogger<CbsClient>
   
   **API Operations:**
   - CreateGroupAccountAsync: POST /api/accounts
     * Creates CBS savings account for stokvel group
     * Returns CBS account number for future transactions
   - CreditAccountAsync: POST /api/transactions (CREDIT)
     * Credits stokvel account (member contributions)
     * Returns transaction ID and updated balance
   - DebitAccountAsync: POST /api/transactions (DEBIT)
     * Debits stokvel account (member payouts)
     * Returns transaction ID and updated balance
   - GetAccountBalanceAsync: GET /api/accounts/{id}/balance
     * Queries current account balance from CBS
     * Returns balance and available balance
   - GetAccountStatementAsync: GET /api/accounts/{id}/statement
     * Retrieves transaction history for date range
     * Returns collection of transactions with dates, amounts, types

2. **Resilience Patterns (Polly 8.6.0):**
   - **Retry Policy:** Exponential backoff with 3 attempts
     * Delays: 2 seconds, 4 seconds, 8 seconds (2^retryAttempt)
     * Retries on non-success HTTP status codes
     * Logs warnings with status code and retry attempt number
     * Policy: WaitAndRetryAsync(3, retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt)))
   
   - **Circuit Breaker:** Opens after 5 consecutive failures
     * Break duration: 1 minute
     * Half-open state after break period for testing
     * Logs error on circuit open, info on reset/half-open
     * Policy: CircuitBreakerAsync(5, TimeSpan.FromMinutes(1))
   
   - **Combined Policy:** Circuit breaker wraps retry (outer → inner)
     * Ensures proper failure isolation
     * Circuit breaker state survives individual retry attempts
     * ExecuteWithResilienceAsync() helper applies combined policy
   
   - **Feature Flags:** EnableRetry and EnableCircuitBreaker in options
     * Allows disabling resilience for testing/debugging
     * Defaults to enabled for production use

3. **Error Handling:**
   - **BrokenCircuitException:** Caught when circuit breaker is open
     * User message: "Core Banking System is temporarily unavailable"
     * Suggests trying again later
   - **HttpRequestException:** Network/transport failures
     * Wrapped in CbsClientException with descriptive message
     * Includes original exception message for debugging
   - **JsonException:** Invalid/malformed CBS responses
     * Wrapped in CbsClientException indicating format problem
   - **Validation:** All responses validated for required fields
     * Throws CbsClientException with specific field missing
   - **Logging:** Error details logged before throwing exceptions
     * Includes operation context, parameters, error details

4. **Configuration (CbsClientOptions):**
   - BaseUrl: CBS API base URL (required, validated on startup)
   - ApiKey: Optional authentication key (sent as X-API-Key header)
   - TimeoutSeconds: HTTP request timeout (default: 30 seconds)
   - EnableRetry: Toggle retry policy (default: true)
   - EnableCircuitBreaker: Toggle circuit breaker (default: true)
   - Bound from appsettings.json section "CbsClient"

5. **Dependency Injection (ExternalServicesDependencyInjection.cs):**
   - AddCbsClient extension method for IServiceCollection
   - Registers ICbsClient → CbsClient with HttpClient factory
   - Configures HttpClient:
     * BaseAddress from options
     * Timeout from options
     * X-API-Key header if ApiKey present
     * Accept: application/json header
   - Binds CbsClientOptions from configuration
   - Validates BaseUrl is provided (throws on missing)
   - Placeholder methods for future clients:
     * AddPaymentGatewayClient (Task 4.2.1)
     * AddSmsGatewayClient (Task 4.3.1)

6. **Internal DTOs:**
   - CreateAccountRequest/Response: Account creation
   - TransactionRequest/Response: Credit/debit transactions
   - BalanceResponse: Balance inquiry results
   - StatementResponse: Transaction history
   - StatementTransaction: Individual transaction details
   - All DTOs internal to CbsClient implementation
   - Not exposed in public API surface

**Technical Notes:**

1. **Retry Strategy:**
   - Exponential backoff prevents CBS overload during failures
   - 3 attempts chosen for balance between reliability and latency
   - Total max latency: ~14 seconds (2s + 4s + 8s) for transient failures
   - Logs each retry with context for debugging

2. **Circuit Breaker Strategy:**
   - 5 failures threshold allows some transient issues without opening
   - 1-minute break gives CBS time to recover from sustained issues
   - Half-open state tests CBS availability before full reconnection
   - Prevents cascading failures to upstream services

3. **Combined Policy Rationale:**
   - Circuit breaker outer, retry inner (Policy.WrapAsync order)
   - Circuit breaker tracks cumulative failures across retries
   - Retry doesn't waste attempts when circuit is open
   - Proper failure isolation and cascading failure prevention

4. **Error Message Design:**
   - User-friendly messages for application layer
   - Technical details logged but not exposed to users
   - Inner exception preserved for debugging
   - Operation context included in all error logs

5. **Configuration Design:**
   - Options pattern follows .NET best practices
   - Validation on startup (fail fast if misconfigured)
   - Feature flags allow disabling resilience for testing
   - Sensible defaults for production use
